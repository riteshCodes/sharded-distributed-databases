\chapter{Implementation}\label{ch:implementation}
%************************************************
This chapter discusses the implementation of a middleware solution as a prototype that provides sharding functionality across multiple Redis database instances, enabling a system of sharded distributed databases. The middleware serves as an intermediary between clients and the associated Redis databases, ensuring efficient and consistent data distribution across the databases. \\

\noindent This chapter begins by outlining the infrastructures and key components of the middleware, communication approach between the clients, the middleware, and the associated Redis databases in sections \ref{techStack}, \ref{4.2}, and \ref{4.3}. Furthermore, sections \ref{4.3} and \ref{4.4} describe the approach of consistent hash-based sharding employed for data sharding and managing in the distributed databases.


\section{Technology Infrastructure}
\label{techStack}
%%\glsresetall % Resets all acronyms to not used
%***************************************************************************
The middleware, a software intermediary sitting between the clients and the associated databases, is implemented in Python 3.9. The middleware connects to and uses Redis (REmote DIctionary Server) \cite{redis}, an open-source NoSQL in-memory key-value store, as the underlying database. The middleware uses the Redis Python interface (redis-py) \cite{redisPY} as a communication interface with the database, which exposes an API (Application Programming Interface) for setting, updating, getting, and deleting key-value pairs. Furthermore, the middleware utilizes a service listener from Group Remote Procedure Call (gRPC) \cite{grpc}, an open-source Remote Procedure Call (RPC) framework, for communication with clients. gRPC facilitates efficient communication between various services in a distributed system and uses Protocol Buffers (Proto) as the Interface Definition Language (IDL) and the underlying message interchange format \cite{wang1993grpc}. This provides a language-agnostic and platform-independent approach to define and call remote services from the middleware \cite{chamas2017comparing}. \\

\noindent Python 3.9 is utilized to implement clients, which interact and test the functionalities of the middleware. Since the communication between the client and the middleware is implemented via the gRPC protocol, clients must implement a gRPC stub, responsible for packaging the request in a Proto message format (serialization), sending the request to the middleware, and unpacking and decoding the received response from the middleware into a client-supported data format (deserialization). \cref{communication} presents a conceptual representation of the communication interfaces within the middleware-supported sharded distributed database systems.

\section{Redis Key-Value Stores as Distributed Databases} \label{4.2}
This thesis focuses on using multiple standalone Redis databases as centralized databases, which together with the middleware forms a system of distributed databases. Redis is an open-source key-value store that operates in-memory and can function as a database, cache, message broker, streaming engine, and furthermore offers data persistence \cite{redis}. The middleware creates a system of homogeneous distributed databases by employing multiple instances of Redis databases. However, each Redis instance is self-sufficient, operating independently without any knowledge of presence of other database nodes. \\

\noindent As the middleware implemented in Python, redis-py, a Python communication interface to Redis key-value stores, is utilized to instrument middleware functionalities. The functions $HSET$ , $HGETALL$ and $DEL$ are leveraged from the Redis server API. A more comprehensive examination of these functions is discussed in section \ref{4.3}. \\

\noindent For performing bulk requests, middleware employs redis-py's pipelines, which buffer multiple commands into a single request to the server. This approach reduces the number of back-and-forth request and response messages between the middleware and the Redis database servers and significantly increases the performance of executing multiple requests \cite{gutierrez2017messaging}. \\

\noindent Further, the middleware employs core principles of Redis transactions, enabling the execution of multiple requests atomically using $MULTI$ and $EXEC$. This guarantees that the multiple interrelated requests are executed as a single isolated operation, ensuring atomicity \cite{gutierrez2017messaging}. Since the middleware uses non-relational databases (Redis key-value store) to ensure a system of distributed database, it can only ensure two out of the three guarantees (consistency, availability, and partition tolerance) as defined in Brewer’s CAP theorem \cite{gilbert2012perspectives, jatana2012survey}. Hence, the implementation of middleware-assisted sharded distributed databases in this thesis can only guarantee eventual consistency, which comes at the cost of high availability and partition tolerance.

\section{Middleware’s Implementation and Operations} \label{4.3}
\begin{figure} [H]
             \ffigbox{\includegraphics[scale=0.45]{gfx/implement/Operation_Overview.pdf}}{\caption{Abstract Overview of Middleware Functionalities}\label{m_function}}

Upon receiving requests with their respective arguments or datasets from clients, the middleware executes consistent hash sharding based on the provided inputs. To efficiently manage these requests, the middleware utilizes Redis database's pipeline approach to build pipelines that relay multiple sub-requests derived from the main request to the appropriate shards and database nodes. Each database node processes the assigned requests individually, forwards the corresponding result to the middleware. The middleware then merges all incoming partial results from the associated databases and sends the aggregated result to the clients. The middleware exposes an API to the connected clients, enabling them to perform four primitive operations: $SET$, $GET$, $GET$-$RANGE$, and $DELETE$.  \cref{m_function} shows an overview of the functionalities of the middleware.\\

\noindent To improve performance, all operations are executed using pipelines provided by redis-py. This allows multiple requests to be issued at once without waiting for the response to each individual request \cite{da2015redis}. Bulk requests from the operations are queued using the pipeline instance from Redis and are only executed once the pipeline is triggered and after all requests are sent to the respective database. This significantly reduces the round-trip times of requests and responses, especially when there are multiple requests \cite{gutierrez2017messaging}. The operations of the middleware used in the system of sharded distributed databases are described below:  

\begin{itemize}

\item[-] $SET:$ \\
This operation writes or updates to single or multiple key-value pairs in the Redis databases. The $SET$ operation accepts either a single key and its corresponding value or a list of key-value pairs as input parameters, depending on if the operation is a single insertion or a bulk insertion in the Redis database. Keys in the Redis databases are unique identifiers \cite{da2015redis}. If the key does not exist, the operation will insert a new entry with the given key-value pair and if the key already exists in Redis, the set operation will update the key’s associated value. Both variations of $SET$ utilize $HSET$ function provided by Redis, which assigns or updates the specified fields to their corresponding values in the hash stored at a key. The result of the operation is either a string ‘OK’ message, indicating successful insertion or update in the Redis database or an error code (‘Key Error’). \section{Consistent Hash Sharding in Redis Databases via Middleware} \label{4.4}
To prototype a system of distributed databases and implement sharding, the middleware uses a consistent hash sharding approach. This method is known for its ability to uniformly distribute data among multiple database nodes and minimize the impact of data redistribution when nodes are added or removed from the system \cite{9004233, karger1997consistent}. This thesis focuses primarily on the data distribution aspect of this algorithm. \\

\noindent Consistent hashing is a hashing approach that operates independently of the number of database servers or data to hash \cite{karger1997consistent}. This approach distributes data and objects across an abstract circle, often referred to as a hash ring, utilizing a hash function and a modular (modulo) operation. In order to allocate an object or data to one of the n database servers, the data must first be hashed using a hash function. The resulting hash value (digest) is then subjected to a modular operation with the number of database servers (n), ensuring that the hash value ranges from 0 to n, thus forming a circular path. Data items placed within this space are sorted in ascending order along the circular path. The search proceeds in a clockwise direction, comparing the hash values of the data and the associated database nodes. The appropriate database node is identified by selecting the node situated immediately after the data's hash value on the hash ring. For instance, if $D_{1}$ represents a dataset and $N_{1}$ represents a database node identifier, $D_{1}$ is placed at N1 if and only if the hash value of $N_{1}$ directly succeeds that of $D_{1}$ within the circular hash ring. \cref{consistent-illus} shows an overview of consistent hashing approach. \\

\noindent For addressing the skewed distribution of data across database nodes, consistent hashing also proposes the use of virtual nodes. Virtual nodes help mitigate the impact of data skew to a certain limit \cite{silberschatz2011database, venkateswaran2017handling}. In the virtual nodes approach, the number of virtual nodes is typically multiple times greater than the number of actual database nodes \cite{silberschatz2011database}. These virtual nodes are consistently mapped to the actual database nodes and distributed across the circular hash ring randomly, ensuring higher probability of achieving uniform data distribution and a lower likelihood of encountering data hotspots when adding or removing database nodes \cite{silberschatz2011database, venkateswaran2017simplified}.

\begin{figure} [H]
             \ffigbox{\includegraphics[scale=0.3]{gfx/implement/Consistent_Hashing.pdf}}{\caption{Illustration of Consistent Hash-based Sharding}\label{consistent-illus}}
\end{figure}

\noindent For concrete implementation, connection identifiers, i.e., Uniform Resource Locator (URL) of Redis database servers are used as database node identifiers. These identifiers are hashed using the Message-Digest (MD5) algorithm \cite{rivest1992rfc1321}, which generates 128-bit hash values.  The MD5 algorithm is suitable for non-cryptographic purposes and for data partitioning in distributed databases due to its randomness in output and low hash collision rate \cite{rivest1992rfc1321}. \\ 

\noindent To reduce the computational overhead associated with large 128-bit values, the hash digest is truncated to 32 bits. This truncated hash digest undergoes again a modulo operation with the maximum 16-bit unsigned integer value, resulting in values ranging from 0 to 65535. This creates a virtual hash ring with a key space spanning of the range: $\bigl[0, (2^{16} - 1)\bigr]$. Data sharding is conducted based on the shard key as specified by the client. The shard key corresponds to the dataset key, which stores values in the Redis database. The shard key is then hashed with the MD5 algorithm, generating a 128-bit hash value. This value is truncated to a 32-bit hash value and then subjected to a modulo operation with the maximum 16-bit number (65535). The first database node with a position greater than the hashed key is assigned the given key-value pair. To improve search efficiency, the appropriate database node search process is done via Binary Search Tree (BST) algorithm. The total number of database nodes with virtual nodes and shards are configurable. For simplicity, each database node represents a single shard of data. The total number of database nodes and their corresponding virtual nodes are configured in the chapter \ref{evaluation} accordingly for the test scenario of data distribution.
